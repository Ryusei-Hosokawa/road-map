## 2. 最終ゴール（10ヶ月後のスタートライン到達）

### ゴール定義

**既存のSvelteKit + TypeScriptプロジェクトにおいて、振られたフロントエンドの開発タスクをほぼ自力で完遂できる。**


### ゴールの具体像

| # | タスク例 | 具体的に何ができるか |
|---|---|---|
| 1 | UIコンポーネントの新規作成 | 仕様を受け取り、SvelteKit + TypeScript + Tailwind CSSで型安全なコンポーネントを実装しPRを提出できる |
| 2 | 既存機能のバグ修正 | バグ報告を受けて、原因調査（DevTools・ログ・コードリーディング）→ 修正 → テスト追加 → PRまで一人で回せる |
| 3 | テストの作成・保守 | Vitestでユニットテスト、PlaywrightでE2Eテストを書き、CIを通過させられる |
| 4 | API連携の実装 | APIドキュメントを読み、フロントからのデータ取得・送信を型安全に実装できる |
| 5 | コードレビューへの対応 | レビュー指摘の意図を理解し適切に修正できる。小タスクは1往復・中タスクは2往復以内でapprovalを得られる |
| 6 | 開発環境の自律的な運用 | Docker環境の起動・停止、Gitブランチ運用、CI結果の確認を自力で行える |

### 達成判定の4軸（定量基準）

**軸1: 独立性**
- 仕様・デザインを受け取り、実装方針を自分で決めて着手できる（方針確認の質問が0回）
- 不明点はドキュメント・コード・GitHubで自力調査してから質問できる
- 作業中の質問が「どうすればいいですか」から「〇〇と理解していますが合ってますか」に変わっている

**軸2: PR品質（0/1で客観的に判定可能）**

| タスク規模 | 定義 | レビュー往復の上限 |
|---|---|---|
| 小 | 既存コンポーネントの修正・スタイル変更 | 1往復以内でapproval |
| 中 | 新規コンポーネントの作成 | 2往復以内でapproval |

PR提出時の必須条件（全て0/1で判定）:
- `tsc --noEmit` が通る（型エラー0）
- lintエラー0
- ブラウザのコンソールにエラーなし
- テストが全てパス

**軸3: テスト**
- 自分が新規実装したロジックにVitestのユニットテストを書ける（0/1）
- Playwrightで主要な操作フローのE2Eテストを少なくとも1シナリオ書ける（0/1）

**軸4: 成果物（アウトプットアプリ v5 の完成）**

| 条件 | 判定 |
|---|---|
| SvelteKit + TypeScript + Tailwind CSSで構築されている | 0/1 |
| CRUD の4操作が全て動作する | 0/1 |
| 認証（ログイン・ログアウト）が実装されている | 0/1 |
| Vitestのユニットテストが含まれ、CI上でパスする | 0/1 |
| PlaywrightのE2Eテストが含まれ、CI上でパスする | 0/1 |
| GitHub Actionsで CI が設定されている | 0/1 |
| GitHubにpublicリポジトリとして公開されている | 0/1 |
| READMEにセットアップ手順・機能一覧・アーキテクチャ説明がある | 0/1 |

全8条件のうち7/8以上で達成とする。

## 3. ゴールから逆算したスキルツリー

最終ゴールを達成するために必要な能力を分解する。
各末端項目に**達成基準（＝観測可能な行動・成果物）**を設定する。

### 3-1. Web / ネットワーク基礎（最重要）

なぜ: エンジニアとしての根幹知識。Web/Networkを理解していないと、バグの原因切り分け・API連携・フレームワークの「なぜ」が一切腑に落ちない。他の技術はあくまで問題解決の一手段に過ぎず、この基礎が全ての学習の土台になる。**Stage 1で最も時間を割く領域。**

```
Web / ネットワーク基礎
├── Webの仕組み
│   ├── クライアント・サーバーモデル
│   │   └── 達成基準: URLを入力してからレスポンスが返るまでの流れを5ステップ以上で図示できる
│   ├── HTTP/HTTPS（メソッド、ステータスコード、ヘッダー、Cookie）
│   │   └── 達成基準: DevToolsのNetworkタブで任意のリクエストを選び、メソッド・ステータス・ヘッダーの意味を説明できる
│   └── DNS・ドメイン・IPアドレス
│       └── 達成基準: nslookupで任意のドメインのIPを調べ、DNS解決の流れを説明できる
│
├── ネットワーク基礎
│   ├── TCP/IP（4層モデル）
│   │   └── 達成基準: 4層それぞれの役割を説明し、HTTPがどの層で動くかを示せる
│   └── ポート番号
│       └── 達成基準: 80/443/3000/5173等の代表的なポートの用途を説明できる
│
├── ブラウザの仕組み
│   ├── レンダリングパイプライン（パース→DOM→CSSOM→レイアウト→描画）
│   │   └── 達成基準: HTML/CSS/JSがブラウザでどの順で処理されるかを図で説明できる
│   └── DevTools活用（Elements, Console, Network, Application）
│       └── 達成基準: DevToolsの4タブで目的の情報に3分以内にたどり着ける
│
├── REST API
│   ├── RESTの原則とHTTPメソッドとCRUDの対応
│   │   └── 達成基準: CRUD操作それぞれに対応するHTTPメソッドとURLパターンを設計できる
│   └── ツールを使ったAPI検証（Postman / curl）
│       └── 達成基準: Postmanで任意の公開APIにGET/POSTリクエストを送り、レスポンスを解釈できる
│
└── セキュリティ基礎
    ├── 主要な脆弱性（XSS, CSRF）
    │   └── 達成基準: 脆弱なコードと対策済みコードを両方実装し、違いを説明できる
    ├── CORS
    │   └── 達成基準: CORSエラーが発生した際に原因を特定し、どこを設定すれば解決するか説明できる
    └── 認証・認可の基本（Cookie, Session, Token）
        └── 達成基準: Cookie認証とToken認証の違いを説明でき、各方式の基本的な流れを図示できる
```

### 3-2. TypeScript

なぜ: 開発プロジェクトの主要言語。コードの読み書きすべてに必要。JSを別途学ぶのではなく最初からTSとして学ぶ。

```
TypeScript
├── 基礎
│   ├── 開発環境構築（tsc, tsconfig.json, エディタ設定）
│   │   └── 達成基準: TSプロジェクトを0からセットアップし、コンパイル・実行できる
│   ├── 基本型・変数・スコープ
│   │   └── 達成基準: any を使わず型注釈付きの変数宣言を書け、スコープの動作を説明できる
│   ├── 条件分岐・ループ・関数
│   │   └── 達成基準: 引数・戻り値に型注釈をつけた関数を書き、呼び出し側で型エラーを検出できる
│   ├── 配列操作（map, filter, find, reduce, forEach）
│   │   └── 達成基準: JSONデータ配列から条件に合う要素を抽出・変換する処理を map/filter で書ける
│   ├── オブジェクト操作（分割代入、スプレッド構文）
│   │   └── 達成基準: ネストしたオブジェクトから分割代入で値を取り出し、スプレッドで新しいオブジェクトを生成できる
│   ├── interface と type
│   │   └── 達成基準: APIレスポンスの型を interface で定義し、コード内で使用できる
│   └── 型推論
│       └── 達成基準: 型注釈を省略してよい場面とすべきでない場面を判断できる
│
├── 応用
│   ├── ジェネリクス（基礎）
│   │   └── 達成基準: 型引数を持つ汎用関数（例: APIレスポンスのラッパー）を定義・使用できる
│   ├── ユーティリティ型（Partial, Required, Pick, Omit, Record）
│   │   └── 達成基準: 既存の型から派生型を作り、フォームの部分更新などに適用できる
│   ├── 型ガード（typeof, in, ユーザー定義型ガード）
│   │   └── 達成基準: union型のレスポンス（成功/エラー）を型ガードで安全に分岐処理できる
│   ├── 非同期処理（Promise, async/await）
│   │   └── 達成基準: async関数でAPIからデータを取得し、try/catch のエラーハンドリングを含めて実装できる
│   └── モジュール（import/export）
│       └── 達成基準: 機能ごとにファイルを分割し、import/export で連携する構成を作れる
│
└── 実務レベル
    ├── 既存コードの型定義を読み解ける
    │   └── 達成基準: 会社のプロジェクトの型定義ファイルを読んで、データ構造を説明できる
    └── 型エラーを自力で解消できる
        └── 達成基準: tsc が出す型エラーメッセージを読み、原因を特定して修正できる
```

### 3-3. Svelte / SvelteKit

なぜ: 会社のプロダクトで使用しているフレームワーク。UIの実装に直結。

```
Svelte / SvelteKit
├── Svelte基礎
│   ├── コンポーネント構造（script / markup / style）
│   │   └── 達成基準: 単一コンポーネントを作成し、TS・テンプレート・スタイルの役割を説明できる
│   ├── リアクティビティ（$state, $derived, $effect）
│   │   └── 達成基準: カウンターアプリを $state + $derived で実装できる（何も見ずに15分以内）
│   ├── Props（親→子へのデータ受け渡し）
│   │   └── 達成基準: 親コンポーネントから受け取った props で表示を切り替えるコンポーネントを作れる
│   ├── イベント処理
│   │   └── 達成基準: ボタンクリック・フォーム送信のイベントをハンドリングし、状態を更新できる
│   ├── テンプレート制御（{#if}, {#each}, {#await}）
│   │   └── 達成基準: 配列データのリスト表示、条件付き表示、ローディング状態の出し分けを実装できる
│   └── バインディング（bind:value 等）
│       └── 達成基準: フォーム入力とコンポーネント状態を双方向バインディングで連携できる
│
├── SvelteKit
│   ├── ファイルベースルーティング
│   │   └── 達成基準: +page.svelte, +layout.svelte, +error.svelte の役割を説明し、3ページ以上のアプリを構築できる
│   ├── データ取得（load関数: +page.ts, +page.server.ts）
│   │   └── 達成基準: load関数でAPIからデータを取得し、ページコンポーネントに渡す実装ができる
│   ├── フォームアクション（form actions）
│   │   └── 達成基準: フォーム送信→サーバー処理→結果表示の一連の流れを実装できる
│   ├── レイアウトとエラーハンドリング
│   │   └── 達成基準: 共通レイアウトとエラーページを実装し、意図通りに動作させられる
│   └── API Routes（+server.ts）
│       └── 達成基準: +server.ts で GET/POST エンドポイントを作成し、フロントから呼び出せる
│
└── 実務レベル
    ├── 既存コンポーネントの構造を読み解ける
    │   └── 達成基準: 会社のプロジェクトの任意のコンポーネントを読んで、データフローとイベントの流れを説明できる
    └── 既存プロジェクトに新規コンポーネントを追加できる
        └── 達成基準: プロジェクトの既存パターンに合わせたコンポーネントをゼロから作成し、PRを出せる
```

### 3-4. テスト

├── ユニットテスト（Vitest）
│   ├── Vitestセットアップと基本構文（describe/it/expect）
│   │   └── 達成基準: Vitestをプロジェクトに導入し、純粋関数に対するテストを5つ以上書いて全て通せる
│   ├── 非同期処理のテスト
│   │   └── 達成基準: async関数のテストを書き、成功ケースとエラーケースの両方を検証できる

│       └── 達成基準: 外部依存（API呼び出し等）をモックに差し替えたテストを書ける

│   ├── Playwrightセットアップと基本操作

│   └── テストシナリオの設計と実装
│       └── 達成基準: ユーザー操作フロー（例: 記録追加→一覧確認→削除）のE2Eテストを設計・実装できる
│
└── 実務レベル
    └── CIでテストを実行し、結果を確認できる
        └── 達成基準: PRを出した際にCIのテスト結果を確認し、失敗時に原因を特定して修正できる
```

### 3-6. CSS / Tailwind CSS

```

## 5. Stage構成概要（基盤先行型）

### なぜこの構成か

- **10ヶ月・5Stage・2ヶ月サイクル**: 1年は遅い。10ヶ月で逆算し、分割しやすい単位で設計。
- **2ヶ月サイクル**: 短すぎると各領域が浅くなりすぎる。3ヶ月は長すぎてフィードバックが遅れる。2ヶ月は周囲のエンジニアからフィードバックを得る機会を増やしながら、十分な深度を確保できる単位。
- **基盤先行型**: Web/Networkを最初の2ヶ月で集中的に固める。その上でフレームワークを学ぶことで、SvelteKitのload関数やAPI連携の「なぜ」が最初から腑に落ちる。
- **重み付きスパイラル**: 初期StageほどWeb/Network・TS・Svelte/SvelteKitに時間を集中。Docker・テスト・セキュリティは後半で比重を上げる。

### Stage一覧

| Stage | 期間 | 有効学習時間 | 重点領域 | Stage末に引き受けられるタスク |
|---|---|---|---|---|
| 1 | Month 1-2（3〜4月） | 約357h | Web/Network・TS・Git・DevTools・CLI | 開発環境起動・コードリーディング・軽微なテキスト修正PR |
| 2 | Month 3-4（5〜6月） | 約357h | Svelte・SvelteKit・REST API・Docker・Web応用 | 既存コンポーネントの軽微な修正・簡単なUI変更PR |
| 3 | Month 5-6（7〜8月） | 約357h | SvelteKit深化・TS応用・Vitest・Webセキュリティ・Tailwind | UIコンポーネント新規作成・テスト付きPR |
| 4 | Month 7-8（9〜10月） | 約357h | Playwright・コンポーネント設計・実業務タスク・DB概念 | バグ修正からPRまで・E2Eテスト作成 |
| 5 | Month 9-10（11〜12月） | 約357h | 実業務タスク中心・弱点補強 | 全開発タスクをほぼ自力で完遂 |

### Stage間の見直しルール

- **毎週末**: 今週の達成基準の自己チェック。未達があれば翌週に繰り越し。
- **各Stage終了時**: 次のStageの週単位スケジュールを詳細化する。2つ先のStageの方向性を見直す。
- **Stage 3終了時**: 最終ゴールへの到達見込みを評価し、Stage 4・5を再設計する。

## 12. 備考

### 学習リソースの方針

- **入門・理解**: インタラクティブ教材・書籍を使う（Recursionist・サバイバルTS・書籍等）
- **リファレンス**: 公式ドキュメントを使う（教材としてではなく確認用）
- **有料教材**: 必要に応じてUdemy等も検討する

### このロードマップの前提

- Stage 3〜5は現時点の知識で分解しているため精度が低い。各Stage終了時に再設計する前提
- 学習ペースは実績に基づいて都度調整が必要
