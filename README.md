## 1. 前提情報

### 1-1. 現在のスキル

| 領域 | レベル | 備考 |
|---|---|---|
| HTML / CSS | 実務2年 | マークアップ業務として一人で遂行可能 |
| JavaScript | 基礎〜中級 | アニメーション・インタラクティブUI実装の経験あり。非同期処理・モジュール設計は未経験 |
| TypeScript | 初歩的な自学経験あり | 1年以上前から自学で断続的に触れている。プリミティブ型の概念理解、変数・関数定義時の型注釈構文は把握。interface/type定義の基本構文を理解しており、AIサポート下で実際に使用した経験あり。import/exportによるファイル分割も概念・構造はざっくり把握。ただし実務での使用経験はなく、監督下での開発経験もない |
| PHP / WordPress | 実務2年（制作レベル） | WPの独自テーマを0ベースから構築する業務を遂行。PHPファイル内にHTMLを記述し、for/foreachによるUI要素の複製、配列データを引数として渡してUI差分を制御する等、マークアップ文脈でのプログラミング的な使い方は実務レベルで理解・使用。WP REST APIを利用したヘッドレスCMS構成でのWebサイト実装経験あり（AIサポート下、複数回）。REST APIやヘッドレスCMSの概念はふわっと理解している |
| Git | 基礎 | 個人利用レベル。チーム運用（PR・レビュー・ブランチ戦略）は未経験 |
| Svelte / SvelteKit | 未経験 | ー |
| Tailwind CSS | 実務経験あり | 基本的な利用は可能。Config設計・v3/v4差分・ライブラリ連携は未経験 |
| テスト | 未経験 | 自動テスト・テスト設計ともに未経験 |
| Docker | 未経験 | ー |
| Web/ネットワーク基礎 | 薄い | HTTP通信やDNSの仕組みなど体系的に学んだことがない |

**プログラミング基礎概念の理解状況（PHP/TS経験を通じて既に持っているもの）:**

以下の概念は、PHP（WP実務）やTypeScript（自学）の経験を通じて、使用文脈レベルでの理解は獲得済み。ただし体系的・言語横断的な理解ではなく、「使った場面での理解」にとどまる。

- 変数、定数
- 条件分岐（if/else）
- ループ（for, foreach/for...of）
- 関数（引数、戻り値の概念）
- 配列とオブジェクト（データの格納・取り出し）
- 型注釈の基本（TS: string, number, boolean等）
- interface / type定義の基本構文
- import / exportの概念
- REST APIの存在と基本的な役割（ヘッドレスCMS経験から）

### 1-2. このロードマップの目的

Web制作会社（コーダー）→ Webアプリ開発会社（フロントエンドエンジニア）へ転職した。
現状、開発業務に貢献できるスキルが不足しているため、業務時間外の自主学習で不足分を補い、できるだけ早く開発タスクを引き受けられる状態になる。

### 1-3. 学習に使える時間

| 区分 | 時間 |
|---|---|
| 平日（月〜金）1日あたり | 3.5h（朝 1.5〜2h + 夜 1.5〜2h） |
| 休日（土日）1日あたり | 12h |
| **週合計** | **41.5h** |
| 月換算（4.3週） | 約178h |
| **年間合計（raw）** | **約2,190h** |
| **年間合計（80%バッファ込み）** | **約1,750h** |

> バッファの理由: 体調不良、予定の変動、想定以上に時間がかかるトピックの存在を考慮し、計画は実学習時間の80%で設計する。

### 1-4. 学習と業務の区別

このロードマップは**業務時間外の自主学習**を対象とする。
業務中のOJT（手動テスト実践、コードレビューを受ける経験など）は別枠であり、このロードマップには含めない。
ただし、自主学習の成果を業務にどう接続するか（＝各Phase終了後に引き受けられるタスク）は明記する。

## 2. 最終ゴール（1年後の到達状態）

### ゴール定義

**既存のSvelteKit + TypeScriptプロジェクトにおいて、フロントエンドの開発タスクを一人で完遂できる。**

### ゴールの具体像（＝このタスクを一人で回せる状態）

> ※想定ベース。CTO確認後に実際の業務タスク例に置き換える。

## 3. ゴールから逆算したスキルツリー

最終ゴール（G1〜G6）を達成するために必要な能力を分解する。
各末端項目に**達成基準（＝観測可能な行動・成果物）**を設定する。

### 3-1. TypeScript（→ G1, G2, G4, G5に必要）

なぜ: 開発プロジェクトの主要言語。コードの読み書きすべてに必要。
JSを別途学ぶのではなく、最初からTSとして学ぶ（JSの基礎概念はTSの中で自然に習得される）。

```
TypeScript
├── 基礎
│   ├── 開発環境構築（tsc, tsconfig.json, エディタ設定）
│   │   └── 達成基準: TSプロジェクトを0からセットアップし、コンパイル・実行できる
│   ├── 基本型（string, number, boolean, null, undefined, array, object）
│   │   └── 達成基準: 各型の特性を説明でき、型注釈付きの変数宣言を書ける
│   ├── 変数・定数（let, const）とスコープ
│   │   └── 達成基準: ブロックスコープの動作を説明でき、適切にlet/constを使い分けられる
│   ├── 条件分岐（if/else, switch, 三項演算子）
│   │   └── 達成基準: FizzBuzzを3通りの条件分岐で実装できる
│   ├── ループ（for, for...of, while）
│   │   └── 達成基準: 配列の全要素を処理する3種のループを書き分けられる
│   ├── 関数（宣言、アロー関数、引数の型、戻り値の型）
│   │   └── 達成基準: 引数・戻り値に型注釈をつけた関数を書き、呼び出し側で型エラーを検出できる
│   ├── 配列操作（map, filter, find, reduce, forEach）
│   │   └── 達成基準: JSONデータ配列から条件に合う要素を抽出・変換する処理をmap/filterで書ける
│   ├── オブジェクト操作（プロパティアクセス、分割代入、スプレッド構文）
│   │   └── 達成基準: ネストしたオブジェクトから分割代入で値を取り出し、スプレッドで新しいオブジェクトを生成できる
│   ├── interface と type
│   │   └── 達成基準: APIレスポンスの型をinterfaceで定義し、コード内で使用できる
│   └── 型推論
│       └── 達成基準: 型注釈を省略してよい場面とすべきでない場面を判断できる
│
├── 応用
│   ├── ジェネリクス（基礎）
│   │   └── 達成基準: 型引数を持つ汎用関数（例: APIレスポンスのラッパー）を定義・使用できる
│   ├── ユーティリティ型（Partial, Required, Pick, Omit, Record）
│   │   └── 達成基準: 既存の型から派生型を作り、フォームの部分更新などに適用できる
│   ├── 型ガード（typeof, in, ユーザー定義型ガード）
│   │   └── 達成基準: union型のレスポンス（成功/エラー）を型ガードで安全に分岐処理できる
│   ├── 非同期処理（Promise, async/await）
│   │   └── 達成基準: async関数でAPIからデータを取得し、エラーハンドリング（try/catch）を含めて実装できる
│   └── モジュール（import/export）
│       └── 達成基準: 機能ごとにファイルを分割し、import/exportで連携する構成を作れる
│
└── 実務レベル
    ├── 既存コードの型定義を読み解ける
    │   └── 達成基準: 会社のプロジェクトの型定義ファイルを読んで、データ構造を説明できる
    └── 型エラーを自力で解消できる
        └── 達成基準: tscが出す型エラーメッセージを読み、原因を特定して修正できる
```

### 3-2. Svelte / SvelteKit（→ G1, G2, G5に必要）

なぜ: 会社のプロダクトで使用しているフレームワーク。UIの実装に直結。

```
Svelte / SvelteKit
├── Svelte基礎
│   ├── コンポーネント構造（script / markup / style）
│   │   └── 達成基準: 単一コンポーネントを作成し、TS・テンプレート・スタイルの役割を説明できる
│   ├── リアクティビティ（$state, $derived, $effect）
│   │   └── 達成基準: カウンターアプリを$state + $derivedで実装できる（何も見ずに15分以内）
│   ├── Props（親→子へのデータ受け渡し）
│   │   └── 達成基準: 親コンポーネントから受け取ったpropsで表示を切り替えるコンポーネントを作れる
│   ├── イベント処理
│   │   └── 達成基準: ボタンクリック・フォーム送信のイベントをハンドリングし、状態を更新できる
│   ├── テンプレート制御（{#if}, {#each}, {#await}）
│   │   └── 達成基準: 配列データのリスト表示、条件付き表示、ローディング状態の出し分けを実装できる
│   └── バインディング（bind:value等）
│       └── 達成基準: フォーム入力とコンポーネント状態を双方向バインディングで連携できる
│
├── SvelteKit
│   ├── ファイルベースルーティング
│   │   └── 達成基準: +page.svelte, +layout.svelte, +error.svelteの役割を説明し、3ページ以上のアプリを構築できる
│   ├── データ取得（load関数: +page.ts, +page.server.ts）
│   │   └── 達成基準: load関数でAPIからデータを取得し、ページコンポーネントに渡す実装ができる
│   ├── フォームアクション（form actions）
│   │   └── 達成基準: フォーム送信→サーバー処理→結果表示の一連の流れを実装できる
│   ├── レイアウトとエラーハンドリング
│   │   └── 達成基準: 共通レイアウトとエラーページを実装し、意図通りに動作させられる
│   └── API Routes（+server.ts）
│       └── 達成基準: +server.tsでGET/POSTエンドポイントを作成し、フロントから呼び出せる
│
└── 実務レベル
    ├── 既存コンポーネントの構造を読み解ける
    │   └── 達成基準: 会社のプロジェクトの任意のコンポーネントを読んで、データフローとイベントの流れを説明できる
    └── 既存プロジェクトに新規コンポーネントを追加できる
        └── 達成基準: プロジェクトの既存パターンに合わせたコンポーネントをゼロから作成し、PRを出せる
```

### 3-3. テスト（→ G2, G3に必要）

なぜ: 品質を担保するための必須スキル。テストが書ける＝自分の実装に責任を持てる。

```
テスト
├── テストの基礎知識
│   ├── テストの目的と種類（単体/結合/E2E）
│   │   └── 達成基準: 各テスト種別の目的・対象範囲・コスト感の違いを図で説明できる
│   └── テスト設計の考え方（正常系/異常系/境界値）
│       └── 達成基準: ある関数の仕様を見て、テストすべきケースを5個以上洗い出せる
│
├── ユニットテスト（Vitest）
│   ├── Vitestセットアップと基本構文（describe/it/expect）
│   │   └── 達成基準: Vitestをプロジェクトに導入し、純粋関数に対するテストを5つ以上書いて全て通せる
│   ├── 非同期処理のテスト
│   │   └── 達成基準: async関数のテストを書き、成功ケースとエラーケースの両方を検証できる
│   └── モック（基礎）
│       └── 達成基準: 外部依存（API呼び出し等）をモックに差し替えたテストを書ける
│
├── E2Eテスト（Playwright）
│   ├── Playwrightセットアップと基本操作
│   │   └── 達成基準: Playwrightをセットアップし、ページ遷移→要素クリック→テキスト検証のテストを書ける
│   └── テストシナリオの設計と実装
│       └── 達成基準: ユーザー操作フロー（例: ログイン→操作→結果確認）のE2Eテストを設計・実装できる
│
└── 実務レベル
    └── CIでテストを実行し、結果を確認できる
        └── 達成基準: PRを出した際にCIのテスト結果を確認し、失敗時に原因を特定して修正できる
```

├── REST API

└── セキュリティ基礎
    ├── 主要な脆弱性（XSS, CSRF）

    └── 認証・認可の基本（Cookie, Session, Token）

### 3-6. CSS / Tailwind CSS（→ G1に必要）

なぜ: 実務経験があるため基礎学習は不要だが、v3/v4差分・Config設計・ライブラリ連携は未経験。
フレームワーク（Svelte）との連携タイミングで深掘りする。

```
Tailwind CSS（深掘り）
├── v3 / v4の差分理解
│   └── 達成基準: v3とv4の主要な違い（CSS-first設定、@theme、新カラーシステム等）を3つ以上説明できる
├── Config設計（theme.extend、カスタムトークン、プラグイン）
│   └── 達成基準: プロジェクトに合わせたtailwind.config.tsを設計し、カスタムカラー・スペーシングを定義できる
└── tailwind-variants等のライブラリ連携
    └── 達成基準: tailwind-variantsで型安全なバリアント（slots, compoundVariants）を定義し、Svelteコンポーネントに適用できる
```

## 10. 備考

### アウトプットの位置づけ

- Phase 1-3: 各学習項目で小さな成果物（コード、図、説明文）を都度作成。達成基準の確認手段を兼ねる

### 学習リソースの方針

- 入門・理解にはインタラクティブ教材や書籍を使う（Recursionist、サバイバルTS、書籍等）
- 公式ドキュメントはリファレンスとして使う（教材としてではない）
